## CRITICAL RULES FOR STEP WRITING

These rules are based on how BAE functional testers actually write test cases. Follow them exactly.

### Keep steps HIGH-LEVEL but SYSTEM-SPECIFIC:
- Each step = one meaningful action or one verification checkpoint
- Multiple related sub-actions CAN be combined into a single step (e.g., "Open Purchase Order from Linked Requisition Line Data. Submit PO for approval")
- Do NOT include session codes in steps (testers know these)
- Do NOT include specific data values unless the test is validating a specific item type or condition. When a specific item prefix or type matters to the test scenario, name it (e.g., "ZZZCST", "ZZZSCMRO", "ZZZSPL-SUBCON-S09-TOOL"). Otherwise the "data" field should be "None".
- Verification-only steps are valid (e.g., "PRQ is in converted status")
- Do NOT include "Login to ERP" or "Navigate to screen" as a step. Start directly with the first meaningful action.
- ALWAYS reference the specific PO series type when the requisition converts to a PO (137, 197, or 397). Do not just say "Purchase Order" when the PO type is known from the test context.

### Keep expected results SHORT:
- One short declarative statement per step
- Often mirrors the action (Action: "Submit for approval" -> Expected: "Submitted for approval")
- No field-level detail, just the observable outcome
- For BPMS case steps, reference the case outcome (e.g., "BPMS015 case completed", "BPMS015 case rejected")

### Scenario descriptions:
- Write as continuous sentences, each action separated by periods
- Follow the pattern: "[Do X]. [Verify Y]. [Do Z]. Expected result: [final state]"
- Include specific item codes or conditions ONLY when testing a specific scenario
- For BWQ steps, be explicit about the outcome: "Run BWQ and accept requisition. Requisition is converted into a [137/197/397] PO."

### Step count guidance:
- Simple happy path (regression): 5-8 steps
- Subcontracted happy path (regression): 8-12 steps (includes production order setup)
- Complex flow with subcontracting/production orders: 8-15 steps
- Functional test (specific condition): 3-7 steps
- Less is more, testers add detail during execution via screenshots

## TEST COVERAGE RULES

For a change request, generate:

**Functional tests**: Test the specific change or condition described in the ECR. Cover the main scenarios plus edge cases. Name format: "[Item/Process Type]: [Specific Condition]" (e.g., "Subcontracted Requisition: Missing Peg/GI Information", "Multi-line Requisition with ZZZSCMRO Cost Item (Return Ticked)")

**Regression tests**: Test existing happy paths to ensure nothing broke. Name format: descriptive flow name like "Single-line Requisition - 137 PO Conversion". Regression tests MUST cover the FULL end-to-end flow from PRQ creation all the way through to PO approval. A regression test that stops at PRQ approval is incomplete.

**CRITICAL CLASSIFICATION RULES**:
- Regression tests are NEVER rejection/failure scenarios. Regression = "this happy path still works after the change".
- Functional tests = "this new validation correctly rejects bad data" or "this specific new behaviour works as specified".
- If a test expects a rejection, error, or auto-reject, it is ALWAYS a Functional test.

**Always generate exactly 2 regression tests**: one for the specific item type or flow being changed (e.g., subcontracted happy path, cost item happy path) and one for a standard/unaffected flow to prove the change did not break existing functionality.

**CRITICAL**: Every regression test MUST go all the way through to PO approval. The minimum flow is: Create PRQ -> Submit -> BPMS015 case completes -> Buyer Review -> BWQ accept -> PRQ converted -> PO generated (specify series) -> Submit PO -> DOA + Supplier Response -> PO approved. Never generate a regression test with fewer than 6 steps.

Generate EXACTLY: 4 functional tests + 2 regression tests = 6 total. No more, no less.

## SYSTEM-SPECIFIC DETAIL REQUIREMENTS

Generated test cases MUST include the following kinds of system-specific detail where relevant. These are the patterns BAE testers expect to see:

### Item prefixes and types:
- ZZZCST = Cost items (lead to 137 series PO)
- ZZZSCMRO = SCMRO cost items (can be Return or CFM Return)
- ZZZSPL-SUBCON-* = Subcontracted items (e.g., ZZZSPL-SUBCON-S09-TOOL)
- SubConDefault = Subcontracted items that must be linked to production orders
- When testing a specific item type, name the prefix in the step action

### PO series types (determined by requisition type and flags):
- 137 = Standard/Cost Item Order (default PO type)
- 197 = Return Order (when "Return" is ticked on requisition)
- 397 = Return Order - Requisition CFM (when "CFM Return" is ticked)
- ALWAYS specify the PO series in the BWQ conversion step and scenario description when the type is known

### Workflow and case references:
- BPMS015 = Requisition Approval workflow (Bizagi). Reference "BPMS015 case" when describing approval/rejection outcomes.
- BPMS040 = DOA Processing for PO approval
- BWQ case trigger = the event that creates a BWQ case for buyer processing
- Always describe whether a BPMS015 case "completes" (approval) or is "rejected" (validation failure)

### Subcontracted requisition flow (DIFFERENT from standard):
Subcontracted items do NOT start with "Create a purchase requisition." The correct flow is:
1. Create a production order
2. Under Planning Line tab, add subcontracting task (e.g., '341ARW1')
3. Go to Actions > Subcontracting > Generate Subcontracting Documents
4. Locate the purchase requisition listed in the Subcontracting Purchase Document
5. Submit PRQ for approval
6. Complete Buyer Review
7. Release production order
8. Run BWQ and accept PRQ
9. PRQ is in "Converted" status
10. Open purchase order (PO number found in Linked Requisition Line Data)
11. Submit PO for approval
12. Complete DOA and supplier response
13. PO is approved

### Return and CFM Return requisition flags:
- "Return ticked" on requisition -> converts to 197 PO
- "CFM Return ticked" on requisition -> converts to 397 PO
- These flags affect which PO series is generated at BWQ conversion

### BWQ detail level:
- Always specify what happens AFTER BWQ accept: "Run BWQ and accept requisition. Requisition is converted into a [series] PO."
- For EP demand scenarios: reference 'Review Candidate PO' task, planner code 'BIZAGIPO', 'Transfer Order Planning' session where relevant

### Email notifications:
- Auto-rejection scenarios always include an email notification step
- The step should say what the email states (e.g., "Email stating purchase requisition is rejected due to PEG/GI info missing")
- Post-buyer-review rejections also trigger email notifications

## DOMAIN KNOWLEDGE

## 1. SYSTEM CONTEXT

BAE Systems uses Infor LN as their ERP system for procurement operations.
All test cases are managed in JIRA with the X-Ray test management plugin.
Design documentation is stored in Confluence (Air ERP Knowledgebase).

## 2. PURCHASE REQUISITION PROCESS FLOW

### Happy Path (Standard Flow):
1. Employee creates a Purchase Requisition (PRQ) in Infor LN
2. PRQ is submitted for approval -> triggers BPMS015 (Requisition Approval Workflow)
3. Approval workflow processes:
   - Auto-rejection check (system validates mandatory fields/criteria)
   - If passes auto-rejection -> goes to Buyer Review
   - Buyer approves or rejects
   - If approval limits require it -> next approver is selected (DOA escalation)
4. Once approved -> PRQ status = "Approved", BPMS015 case = "Completed"
5. Buyer Work Queue (BWQ) processes the approved requisition
6. BWQ accept -> PRQ status = "Converted" -> Purchase Order (PO) is generated
   - Standard/cost item -> 137 series PO
   - Return ticked -> 197 series PO
   - CFM Return ticked -> 397 series PO
7. PO is submitted for approval -> triggers BPMS040 (DOA processing)
8. DOA approval + supply response -> PO status = "Approved"

### Subcontracted Flow (DIFFERENT from standard):
1. Create production order in LN
2. Add subcontracting task under Planning Line tab (e.g., '341ARW1')
3. Actions > Subcontracting > Generate Subcontracting Documents
4. Locate PRQ in Subcontracting Purchase Document
5. Submit PRQ for approval -> triggers BPMS015
6. Complete Buyer Review -> BPMS015 case completed
7. Release production order
8. Run BWQ and accept -> PRQ converted
9. Open PO from Linked Requisition Line Data -> Submit PO
10. DOA + supplier response -> PO approved

### Auto-Rejection Scenarios (BPMS015):
- Missing peg/GI information for subcontracted items
- Blank BP or EMPloyee supplier on subcontracted items (rejected after buyer review)
- SubConDefault items not linked to production orders
- Multi-line requisitions with subcontracted items
- Multi-line requisitions with ZZZSCMRO cost items (Return or CFM Return)
- When auto-rejection occurs, the requester receives a rejection email notification

### Purchase Requisition Types and PO Series:
- Standard PRQ -> 137 PO (Cost Item Order)
- PRQ with Return ticked -> 197 PO (Return Order)
- PRQ with CFM Return ticked -> 397 PO (Return Order - Requisition CFM)
- Subcontracted PRQ (originates from production order subcontracting documents)

## 7. CHANGE REQUEST (ECR) STRUCTURE IN JIRA

### Fields that matter for test creation:
- **Title**: Brief description of the change
- **Labels**: Business area (e.g., "Procurement", "Buyer Work Queue")
- **Components**: Related business processes (e.g., BPMS015)
- **Description** (structured format):
  - Requirement / Issue: Background context for why the change is needed
  - Exact Change Required: Specific technical/functional change (often has before/after)
  - Security Model Impact: Any security changes
  - Benefit of Change: Business justification
  - Point of Contact / Sponsor: Who requested it
- **PBTs**: Developer work tasks (may contain unit test evidence)

### What testers focus on:
- "Requirement or Issue" section -> understand the WHY
- "Exact Change Required" section -> understand the WHAT
- Components -> identify which business processes are affected
- Labels -> identify the functional area

## 8. X-RAY TEST CASE FORMAT

### Test ticket fields:
- **Summary**: Short descriptive title
- **Type**: Manual
- **Test Type**: Functional or Regression
- **Labels**: Matching the ECR labels
- **Components**: Matching the ECR components
- **Scenario**: Description of what the test covers (the end-to-end flow)
- **Expected Result**: The final expected state after all steps complete
- **Preconditions**: What must be true before the test starts (often left empty)

### Test step structure:
Each step has three fields:
| Field | Description |
|---|---|
| Action | What the tester does |
| Data | Almost always "None" |
| Expected Result | Short outcome statement |

## 9. TEST CATEGORIES

### Regression Tests:
- Cover standard happy path flows
- Verify existing functionality has not broken
- More predictable/templatable
- MUST specify the PO series type at BWQ conversion
- Example: "Create PRQ -> Approve -> Convert to 137 PO -> Approve PO"

### Functional Tests:
- Test the specific change described in the ECR
- Require understanding of what changed (before vs after)
- Cover edge cases, error scenarios, negative paths
- Reference BPMS015 case status in expected results (completed vs rejected)
- Example: "Verify BPMS015 case rejects multi-line requisition with ZZZSCMRO cost item"
